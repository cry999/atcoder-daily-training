# ADR 0008: ランダム入力生成はベストエフォート解析 + 生セクションキャッシュで行う

- ステータス: Accepted
- 日付: 2026-07-01
- 要件: [requirements/060-gen-random-input.md](../requirements/060-gen-random-input.md)
- 関連: [requirements/024-interactive-case-builder.md](../requirements/024-interactive-case-builder.md) (追加ケース `tests-extra/`・入力のみケースの保存規約) / [requirements/001-exercise-test.md](../requirements/001-exercise-test.md) (fetch / cache 基盤)

## コンテキスト

問題の **制約 (Constraints)** と **入力形式 (Input)** を認識してランダムテスト入力を生成する機能 (`atcoder gen` / chat `:gen`) を足す。実装可能性の調査で以下が分かった:

- 問題ページ HTML 全体は既に `internal/testexec/fetch.go` の `fetchProblem()` が取得済みで、制約・入力形式節も同じ HTML 内にある (今は time limit とサンプルだけ保存)。HTML パースは `antchfx/htmlquery` が依存に入っている。
- 入力形式節は `<pre>` で半構造化されており構造解析しやすい。制約節の `1 ≤ N ≤ 10^5` 系は正規表現で抽出できる (体感 7〜9 割)。
- 一方、「A は順列」「グラフは連結」「英小文字のみ」「テストケース総和 ≤」「単調増加」などの**自然言語の構造的制約**は正規表現では拾えない。完全な自動認識は原理的に不可能。
- 外部ツール `online-judge-template-generator` が同種の解析で generator を自動生成している実績があり、方式自体は成立する。

ここで 2 つの設計判断が要った: **(1) どこまで自動化するか**、**(2) 解析元をどう永続化するか**。

## 決定

### (1) ベストエフォート即生成に倒す

拾えた制約でその場でランダム入力を生成する。取りこぼした変数は既定レンジ (int `1..10^9` / string 長 `1..10^5`・`a-z`) + **警告**で埋め、構造的制約は無視して独立生成し `coverage=partial` を明示する。**出力の正しさは保証しない下準備ツール**と位置づけ、`--show-spec` で「何を認識できたか」を透明化する。

### (2) 生セクションを source-of-truth としてキャッシュする

解析済みの構造 (`Spec`) ではなく、制約・入力形式節の**生テキスト**を新ファイル `gen.toml` (`[raw]` セクション) に保存し、生成のたびに生テキストから `Spec` を組む。`Spec` は永続化しない中間表現とする。

## 代替案 (却下)

- **完全自動認識を目指す**: 構造的制約 (順列・連結グラフ・木・単調列) まで理解して意味的に妥当な入力だけ生成する。理想だが、自然言語の網羅は青天井で、"認識できたつもりで誤った入力を確信を持って吐く" ほうが黙って間違えるぶん危険。ベストエフォート + 警告のほうが正直で、ユーザ選択 (「ベストエフォート即生成」) とも合致。構造理解は将来の拡張余地として `coverage` を広げる形で足す。
- **generator 雛形を出力する (oj-template 流)**: `Spec` を Python の `generate.py` 雛形にして、ユーザが構造制約を手直しして使う。確実性は高いが一手間かかり、ユーザ選択 (即生成) と外れる。解析と生成を分離してあるので、同じ `Spec` を別レンダラに渡すだけで**後から足せる**。今は作らない。
- **解析済み `Spec` を `gen.toml` に保存する**: 生成のたびの再解析を省けるが、解析ヒューリスティックを改善するたびに**再 fetch が要る**うえ、スキーマが複雑化して手編集しづらい。生テキスト保存なら (a) 再 fetch なしで再解析でき、(b) 将来のストレステスト / 雛形出力も同じ `gen.toml`→`Spec` を入力にできる。追加 HTTP は発生しない (既存 fetch と同じ HTML から拾う) ので、再解析コストは無視できる。
- **`meta.toml` に制約・入力形式を相乗りさせる**: ファイルは増えないが、`meta.toml` はタスクの time limit / URL という平坦な値の器で、複数行の生テキストを混ぜると役割が濁る。別ファイル `gen.toml` に分けて責務を保つ。

## 結果

- 「認識できた範囲で即生成、取りこぼしは警告」という正直な UX に倒せた。`--show-spec` / `coverage` で認識結果が見える。
- 生テキストキャッシュにより、解析器 (`internal/gen`) の改善が再 fetch なしで効く。解析 (`ParseSpec`) と生成 (`Generate`) を分離し、判定ループ (`internal/testexec`) と疎に保ったことで、**ストレステスト**と **generator 雛形出力**を後から低コストで足せる下地ができた。
- 生成物は解答ファイルに触れず、`--refresh` はキャッシュ (`gen.toml`) のみ、exit code 規約 (引数誤り=2 / 実行時失敗=1 / 成功=0) を踏襲する既存の不変則を保つ。
