# チャット入力の複数行ペースト対応 要件定義

## 概要

インタラクティブ chat (`test --interactive` / `start` 分割画面の下ペイン) の入力欄に **複数行をペーストしたとき、各改行を `Enter` と同じ扱いで子プロセスへ逐次送信**できるようにする。現状は単一行 `textinput` のため、複数行を貼り付けても 1 行扱いになり、改行を含む入力 (多くの問題で必要な「N 行の標準入力」) をまとめて流せない。新しいキーやエディタ化はせず、**ペースト時のみ**「改行 = `Enter`」として各行を順に送る (末尾の未確定行は入力欄に残す)。

## 背景・目的

- AtCoder の問題は「1 行目に N、2 行目に数列…」のように**複数行の標準入力**を取るものが多い。chat で試すとき、手元のサンプル入力をそのまま貼り付けて一気に流したい。
- 今は `textinput` (単一行) なので、複数行を貼ると改行が潰れて 1 行になり、1 行ずつ手で打ち直す必要がある。
- 手入力までリッチな複数行エディタにする必要はない (ユーザ意向: **コピペにだけ対応できれば、あとは逐次送信でよい**)。よって**ペーストの複数行だけ**を「改行ごとに送信」に対応させる。

## スコープ

| 項目 | 当面のスコープ | 将来の拡張余地 |
|---|---|---|
| 対象 | chat の **insert モード**の入力欄へのペースト | 手入力での複数行編集 (textarea 化・改行キー) |
| 挙動 | ペースト内の各改行を `Enter` と同義に: 完全な行は子へ**逐次送信**、末尾の未改行テキストは入力欄に残す | ペースト前プレビュー / まとめて 1 ブロック表示 |
| 改行正規化 | `\r\n` / `\r` を `\n` に正規化してから分割 | — |
| 送信単位 | 1 行ごとに `Fprintln` (空行も 1 行として送る = `Enter` 相当) | バッチ最適化 |

### 境界

- **insert モード限定**。command モード (`:` 行) と ケースビルダー (`textarea`) は各々のペースト処理のまま不変。
- `Enter` (単一行送信)・`Up`/`Down` 履歴・`Tab` 補完・`Ctrl+C`/`Ctrl+D`/`Ctrl+S`・出力タイミング・watch-reload・command モード遷移・ナビ (`NavEnabled`) は**すべて不変**。
- 解答ファイル・判定・exit code には触れない (chat の入力経路のみ)。

## 動作仕様

| 状況 | 挙動 |
|---|---|
| 改行を含むペースト (`"5\n1 2 3\n"`) | `5` → `1 2 3` を順に子へ送信 (各 `Fprintln`)、入力欄は空になる |
| 末尾が改行でないペースト (`"5\n1 2 3"`) | `5` を送信、`1 2 3` は**入力欄に残る** (未確定行。`Enter` で送れる) |
| 入力欄に既存テキストがある状態でのペースト | 現在値の末尾にペーストを連結してから分割・送信する (カーソル途中挿入は簡略化し末尾結合とする) |
| 改行を含まないペースト | 通常どおり `textinput` に挿入 (従来の 1 行入力。送信しない) |
| 空行を含むペースト (`"a\n\nb\n"`) | `a` → `(空行)` → `b` を送信 (空行も `Enter` 相当) |
| `\r\n` / `\r` 混在 | `\n` に正規化してから分割 (CRLF のサンプルでも崩れない) |
| 子が未起動 (遅延起動/子終了後) | 最初の行送信を機に `restart()` で (再) 起動 (既存の遅延起動と同じ)。spawn 失敗時は送信しない |
| 送信後の待機表示 | バッチ送信の**最後に 1 回**だけ出力待ちスピナーを起動 (行ごとに多重起動しない) |
| 履歴 | 送信した各行を履歴に積む (空行は積まない。既存 `Enter` と同じ規則) |

- **既存非破壊**: ペースト (改行あり) を行わない限り、入力欄の挙動は完全に従来どおり。

## CLI / TUI 仕様

- **新フラグ・新キー無し**。bubbletea の bracketed paste (`KeyMsg.Paste == true`) で届く複数行ペーストを insert モードで横取りして処理する。bracketed paste 無効な端末では、ペーストが個別キー + `Enter` 列として届くため**既存の `Enter` 経路がそのまま逐次送信**する (二重対応で取りこぼさない)。

### 処理ステップ (ペースト受信時)

1. insert モードで `KeyMsg` が **paste** (`msg.Paste`) か、runes に改行を含む場合に横取りする。
2. `現在の入力値 + ペースト文字列` を組み、`\r\n`/`\r` を `\n` に正規化。
3. `\n` で分割。最後の要素以外 (= 改行で終わった完全な行) を順に `submitLine` で子へ送る。
4. 最後の要素 (未改行テキスト) を入力欄の新しい値に設定 (`CursorEnd`)。
5. 1 行でも送ったら、最後に出力待ちスピナーを 1 回起動。

### 出力イメージ (chat)

```
» (ここに 3 行貼り付け: "3\n1 2 3\nquery\n")
   → 3
   → 1 2 3
   → query
   2ms ← ...        ← 子からの応答
```

## エラーハンドリング

| 状況 | 動作 |
|---|---|
| 子への書き込み失敗 (`Fprintln` エラー) | その行で `(write failed: …)` を 1 行出し、残りの行は送らない (既存 `Enter` と同じ) |
| spawn 失敗 (遅延起動で子を起こせない) | 送信せず終了 (`restart` が `tea.Quit` を返す既存挙動) |
| exit code | 影響なし (chat 内の入力処理のみ。引数誤り=2 / 実行時失敗=1 は不変) |

## 非機能要件

- **既存非破壊**: `Enter`・履歴・補完・command モード・キー操作・描画は不変。複数行ペースト以外は従来どおり。
- **stdout 非汚染**: 送信した行は chat 内 `kindIn` 行として表示するだけ (stdout には子の出力以外を書かない)。
- **端末互換**: bracketed paste 有効/無効の双方で取りこぼさない (有効=`Paste` 横取り、無効=`Enter` 列)。`\r\n`/`\r` を正規化。
- **決定的にテスト可能**: 分割は純粋関数 `splitPasteLines` に隔離し、CRLF・末尾改行・空行・改行なしをユニットテストで固定する。

## 将来の拡張ポイント

- 手入力での複数行編集 (textarea 化 + `Ctrl+J` 改行)。今回はペーストのみ対応で、必要になればこの要件を土台に拡張する。
- 大量行ペーストのバッチ送信最適化・送信前プレビュー。

## 用語

- **ペースト (bracketed paste)**: 端末が貼り付けを 1 イベントにまとめて送る方式。bubbletea では `KeyMsg.Paste == true` で届き、改行を含む runes をそのまま渡す。
- **insert モード**: chat の通常入力 (`Esc` で command モードへ)。
