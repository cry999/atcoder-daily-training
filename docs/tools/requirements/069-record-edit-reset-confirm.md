# 069 record edit — リセット操作の確認

## 概要

`atcoder record edit` / chat `:record edit` の全画面編集フォームで、計測状態 (state) を
**未計測へリセットする操作**に確認ステップを挟む。要件 068 でフォーム内から `停止 → 未計測`
(reset) や `Backspace` による即リセットができるようになったが、リセットは started_at /
solved_at / duration / target / ac / editorial / 5 軸を **一度に全消去する破壊的操作**であり、
1 打鍵で誤発火すると記録済みの計測・評価が失われる。リセット前に「本当にリセットしてよいか」を
確認し、明示的な肯定 (`y`) があったときだけ実行する。

## 背景・目的

- 要件 068 の reset は「停止から未計測へ戻す」サイクルの一手として、また state 行の `Backspace`
  として **確認なしで即座に全クリア**していた (要件 068 の当時判断: 「フォームでは確認を挟めない」)。
- しかし reset は `:record start restart` 相当の完全破棄で、started_at を含む全フィールドを失う。
  トグルの連打や `Backspace` の誤爆で、せっかく記録した計測時間・5 軸評価が一瞬で消える危険がある。
- start / stop と違い reset は**元に戻せない** (un-stamp を提供していない。要件 068 スコープ)。
  破壊的で不可逆な操作にだけ確認を挟むことで、他の非破壊トグルの軽快さは保ちつつ事故を防ぐ。

本要件は要件 068 の「フォームでは確認を挟めない」という決定を **reset に限って更新**する。
フォームは全画面 bubbletea モデルなので、確認プロンプト行と肯定キー待ちの一時状態を持てば
モーダル確認を実装できる (当時の想定を実装可能な形に見直す)。

## スコープ

| 項目 | 当面のスコープ | 将来の拡張余地 |
|---|---|---|
| 確認対象 | **reset (未計測への全クリア)** のみ。state 行の `Backspace` と `停止 → 未計測` トグルの両経路 | 保存 (Enter) や個別フィールド消去への確認 |
| 確認 UI | フォーム内に確認プロンプト行を出し、次打鍵で判定するモーダル確認 | y/n ダイアログの独立ウィンドウ化 |
| 肯定キー | `y` / `Y` のみリセット実行。**それ以外のキーはすべて取消** (安全側) | Enter を肯定に含める等のキー拡張 |
| 非対象トグル | `未計測 →(start)→ 計測中` / `計測中 →(stop)→ 停止` は破壊的でないため**確認なしで即実行** | — |
| 起動経路 | CLI `record edit` (standalone) と chat `:record edit` (埋め込み) の双方 (フォーム共通実装) | — |

reset 以外の操作 (start / stop トグル、ac/editorial/duration/5 軸の編集、保存、取消) の挙動は
要件 066 / 068 のまま不変。確認は **reset の 1 点にだけ**足す。

## 確認フロー

reset を引き起こすキー操作は、即実行せず **確認待ち状態 (`pendingReset`)** に入る。

| 起点 | 操作 | 挙動 |
|---|---|---|
| state 行 | `Backspace` | 即リセットせず確認待ちへ入る (プロンプト表示) |
| state 行が `停止` | `Tab` / `space` / `h` / `l` (トグル) | `停止 → 未計測` は reset なので確認待ちへ入る |
| state 行が `未計測` | `Tab` / `space` / `h` / `l` (トグル) | start。**確認なし**で即実行 |
| state 行が `計測中` | `Tab` / `space` / `h` / `l` (トグル) | stop。**確認なし**で即実行 |

確認待ち中の次打鍵で判定する:

| 確認待ち中のキー | 挙動 |
|---|---|
| `y` / `Y` | **リセット実行** (全フィールドを空へ)。確認待ちを解除し通常フォームへ戻る |
| それ以外のすべてのキー (`n` / `Esc` / `Enter` / 移動 / 文字 …) | **取消**。何もリセットせず確認待ちを解除。そのキー本来の操作も行わない (誤操作防止のため打鍵を吸収) |

- 肯定は `y` / `Y` に限定する。曖昧なキー (Enter / space 等) を肯定に含めると、保存や別操作のつもりの
  打鍵で誤ってリセットしうるため、**明示的な `y` のみ**を実行トリガーとする (不確実なら取消、が安全側)。
- 確認待ち中は他の編集キーを吸収する (取消として消費)。確認を跨いで別フィールドが動くことはない。
- 確認待ち状態は取消・実行のいずれでも 1 打鍵で解除され、フォームは通常状態へ戻る。

## フォーム UI

確認待ち中は、フォーム下部に確認プロンプト行を出す (破壊的操作を示す警告色)。

```
record edit  abc457_d

> state       [ 停止 ]  開始 16:00 → 完了 16:25
  ac          [ true ]
  editorial   [ false ]
  duration    [ 25m ]
  knowledge   [ 2 ]
  ...

state をリセットします (開始/完了時刻・duration・目標・ac/editorial・5 軸を全消去)。 y=実行 / 他キー=取消
```

- 確認待ち中はヒント行の代わりに確認プロンプトを表示する。プロンプトは何が消えるかを明示する。
- 確認待ちでない通常時のヒント行・表示は要件 066 / 068 のまま不変。

## 動作仕様

| 状況 | 動作 |
|---|---|
| state 行で `Backspace` | 確認待ちへ (即リセットしない) |
| `停止` で state トグル | 確認待ちへ (即リセットしない) |
| `未計測`/`計測中` で state トグル | 従来どおり start/stop を即実行 (確認なし) |
| 確認待ちで `y`/`Y` | resetState を実行し全クリア。確認待ち解除 |
| 確認待ちでその他キー | 取消。リセットせず確認待ち解除。そのキーの本来動作も行わない |
| 確認待ち中の保存/取消 | Enter/Esc も「その他キー」として取消扱い (確認を先に解く)。フォーム自体の save/cancel は発火しない |
| chat `:record edit` | フォーム共通実装のため同じ確認フローが効く。REC 同期 (要件 068) は保存時のみで確認とは独立 |

## 影響範囲

| ファイル | 変更内容 |
|---|---|
| `internal/ui/recordedit.go` | モデルに `pendingReset bool` を追加。`handleKey` の state 行 reset 経路 (`Backspace`・`停止` トグル) を「確認待ちへ入る」よう変更し、確認待ち中は打鍵を確認判定 (`y`=実行 / 他=取消) に振り向ける。start/stop トグルは即実行のまま。`View` に確認プロンプト行を追加。`toggleState`/`resetState` の純遷移は不変 |
| `internal/ui/recordedit_test.go` | Backspace / 停止トグルが即リセットしなくなる分、既存テストを確認フロー (`y` で確定) に合わせて更新。確認→実行・確認→取消・start/stop は確認なし、のテストを追加 |
| `docs/tools/requirements/068-record-edit-state-toggle.md` | 「フォームでは確認を挟めない」注記を本要件で更新した旨の相互リンクを追記 |
| `docs/tools/usage/record.md` | reset に確認が入る旨を追記 |

`internal/ui` の層境界は不変: 確認はモデル内の一時状態と描画だけで完結し、I/O・layout・config を
呼ばない。`pendingReset` は保存対象ではなくフォームを閉じれば消える純 UI 状態。

新規/変更する ui 内部要素 (公開 API は不変):

```go
type recordEditModel struct {
    // ...
    pendingReset bool // reset 確認待ち (y で resetState 実行、他キーで取消)
}

// requestReset / requestStateAdvance は reset 経路を確認待ちに落とす (start/stop は即実行)。
// resolveResetConfirm は確認待ち中の 1 打鍵を y=実行 / 他=取消 で判定する。
```

## エラーハンドリング

| 状況 | 動作 | exit |
|---|---|---|
| 確認待ち中の任意キー | ローカルなモデル操作。I/O を伴わずエラーを出さない | — |
| リセット実行 | 要件 068 の resetState と同じ (全クリア。I/O なし) | — |

確認フロー自体は I/O を伴わないため exit code に影響しない。保存・fetch 等のエラー規約は
要件 066 / 068 のまま。

## 非機能要件

- **既存非破壊**: reset 以外の操作 (start/stop/編集/保存/取消) の挙動・キーは不変。確認は reset の
  1 経路にだけ足す。トグルしなければ時刻は従来どおり保全される (要件 068)。
- **安全側デフォルト**: 肯定は `y`/`Y` に限定し、曖昧・想定外のキーはすべて取消に倒す。誤爆で
  破壊が起きるより、明示確認を要求する方を選ぶ。
- **層境界**: `internal/ui` は layout/config/testexec/file I/O を知らない。`pendingReset` は
  純 UI 状態で保存されない。
- **前方互換**: スキーマ不変 (新キーを増やさない)。確認は表示・入力処理のみで Stat には現れない。
- **既存挙動との整合**: リセット後の結果は要件 068 の reset と同一 (全クリア)。確認は「実行するか
  否か」を挟むだけで、実行された reset の意味論は変えない。

## 用語

要件 061/064/066/068 に準拠 (`contest_id`=`abc457` / `task_id`=`abc457_d` / `letter`=`d` /
`category`=`abc`)。「reset」は state 行から `停止`/任意状態を `未計測` へ戻す全クリア操作。
