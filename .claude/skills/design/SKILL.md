# design

`atcoder` CLI の変更を **設計だけ** で進めるためのスキル。この repo は「要件定義を文章化してから実装する」流儀で育てている (`feature` の手順 1)。design はその **設計フェーズだけを単体で回す** もの: スコープを掴み、設計判断をして、要件定義ドキュメントを書き、ロードマップに載せ、docs コミットとして残す。**実装 (コード・fixture・利用ドキュメント) には踏み込まない。**

「まず仕様を固めたい」「着手前に設計をレビューしてほしい」「複数の実現案を比較して決めたい」ときに使い、実装はあらためて `feature` に引き継ぐ。

## いつ使うか

- 新機能・新フラグ・新モードを **作る前に仕様を文章化** したい (要件定義だけ先に書く)。
- 実現案が複数あり、**代替案・トレードオフを比較して方針を決めたい** (設計レビュー / 意思決定)。
- ロードマップ項目 (`todo.md` / `abc-todo.md`) を「設計に落とす」が、実装はまだしない。
- 大きめの変更で、**前方互換や影響範囲を先に詰めておきたい** (後から E 本番モード判定 / G タイマーが `contest.toml` 等を入力にできるよう)。
- 既存の決定を見直して **ADR (`docs/tools/decisions/`) に決定記録を残したい**。

## いつ使わないか (→ 別の道)

- そのまま **実装まで進めたい** → `feature` (要件定義 → 実装 → fixture → 利用ドキュメント → マージ)。design で書いた要件をそのまま入力にできる。
- typo・コメント・練習解答追加など **仕様を伴わない軽微作業** → `smallwork`。
- 実装とドキュメントの **ズレを直す整頓** → `maketidy`。
- どの流儀か **迷う** → `triage` (smallwork / feature の振り分け入口)。
- 読むだけ・調べるだけ (成果物を残さない) → スキル不要。

## 成果物

design の納品物は **要件定義ドキュメント** (`docs/tools/requirements/NNN-<name>.md`) と、必要なら **ロードマップ更新** / **ADR**。コードや fixture、利用ドキュメントは書かない。「何を・なぜ・どう作るか」を文章で確定させ、実装は別途 `feature` で行う。

## ワークフロー

### 0. worktree を切る (CLAUDE.md 必須)

要件ドキュメントもリポジトリにコミットする成果物なので、変更は worktree で行う。設計内容を表すブランチ名で main から切る。

```sh
git worktree add ../atcoder-daily-training.worktrees/<branch> -b <branch>
```

ブランチ名は `design-<topic>` 形が分かりやすい (`design-judge-cache`, `design-go-runner`)。worktree 内のファイルは絶対パスで Write/Edit する (`cd` は permission prompt を誘発しやすい)。コミットは `git -C <worktree-path> ...`。

### 1. スコープと現状を掴む (読み取りのみ)

設計対象の周辺を調べる。**ここでコードは変えない。**

- 既存の要件 (`docs/tools/requirements/`) とロードマップ (`todo.md` / `abc-todo.md`)、決定記録 (`docs/tools/decisions/`) を読み、重複・前例・分担境界を確認する。
- 触れることになる実装 (`cmd/atcoder/` + `internal/`) の現状を把握する。広く探すなら `Explore`、設計の選択肢出しには `Plan` エージェントを使ってよい (どちらも読み取り専用)。
- 既存の不変則を設計の入力にする: 解答ファイルを壊さない / `--refresh` 系はキャッシュのみ / exit code 規約 (引数誤り = 2・実行時失敗 = 1・成功 = 0) / 標準 `flag` のまま (FW 非導入)。

### 2. 設計判断をする (代替案・トレードオフ)

- 実現案が複数あるなら **案を並べ、利点/欠点/コストを比較** して 1 つに倒す。「却下した代替案」は要件 or ADR に残す価値がある。
- **本当にユーザに決めてもらうべき分岐** (CLI 表面の命名、対象範囲、互換性の割り切りなど、コードからは決まらないもの) は `AskUserQuestion` で確認する。慣例的な既定があるものは決めて進め、判断を述べる。
- 前方互換を意識する。「軽いと思ったら重かった」を避けるため、境界 (他項目との分担) と将来の拡張余地を明示する。

### 3. 要件定義を書く — `docs/tools/requirements/NNN-<name>.md`

既存の連番に続けて 3 桁ゼロ埋め (`ls docs/tools/requirements/` で最新番号を確認してから採番)。`003-exercise-abc-contest-meta.md` を雛形にすると速い。構成:

- **概要** — 1 段落で「何を 1 コマンド/1 フラグで済ませたいか」。
- **背景・目的** — 今のフリクション、なぜ要るか。
- **スコープ** (表) — 当面のスコープ / 将来の拡張余地。境界 (他項目との分担) を明記。
- **ディレクトリ構造 / スキーマ** — 触る/作るパス、TOML 等のスキーマを表で。
- **CLI 仕様** — 引数・フラグ表、処理ステップ (番号付き)、出力イメージ (コマンド例とその stdout)。
- **動作仕様** — 冪等性・部分更新・既存ワークフローとの共存など、表で網羅。
- **影響範囲** (表) — ファイルごとの変更内容。新規パッケージは責務と公開 API シグネチャを Go で素描 (実装はしないが、設計として API 形だけ描く)。
- **エラーハンドリング** (表) — 状況 → 動作 (exit code を含む。引数/フラグ誤りは 2、実行時失敗は 1)。
- **非機能要件** — 冪等性・既存非破壊・前方互換・rate limit 配慮 など。
- **将来の拡張ポイント / 用語 / 関連ドキュメント**。

文章は日本語。ID 用語 (`contest_id`=`abc457` / `contest_num`=`457` / `task_id`=`abc457_d` / `letter`=`d`) は既存要件に合わせる。関連する要件には相互リンクを張る。

### 4. ロードマップ / ADR を更新する

- ツール全般 → `docs/tools/todo.md`、ABC 本番系 → `docs/tools/abc-todo.md` に項目を足す/進め、本要件へ相互リンクする (ロードマップ ⇄ requirements)。
- 既存の決定を見直した・大きな方針を確定したなら、`docs/tools/decisions/NNNN-*.md` (ADR) に決定記録を残し、要件から相互リンクする。完了済み機能の追加設計なら、元 ADR に「後続」リンクを足す。

### 5. コミットして main へ ff-merge する

設計成果物は **docs スコープ** でコミットする。例: `docs(tools): write judge-cache requirements (013)`、`docs(tools): record go-runner ADR`。メッセージ末尾に環境指定の `Co-Authored-By` trailer を付ける。

```sh
git merge --ff-only <branch>
git worktree remove ../atcoder-daily-training.worktrees/<branch>
git branch -d <branch>
```

### 6. 実装は feature へ申し送る

design はここで止まる。実装したくなったら、書いた要件をそのまま入力に `feature` を起動する (手順 1 の要件定義は design で済んでいるので、ロードマップ確認 → 実装 → fixture → 利用ドキュメント → マージ に進める)。「実装はしていない / これは設計のみ」と明示して締める。

## 注意

- **実装に踏み込まない。** コード (`cmd/atcoder/` + `internal/`)・fixture・利用ドキュメント (`atcoder-*-usage.md`) は書かない。書きたくなったらそれは `feature` 案件。
- 要件は「先に文章で仕様を固める」repo の流儀の中核。後から本番モード判定 / タイマー等が `contest.toml` 等を入力にできるよう、前方互換を意識して設計する。
- 設計に不変則を織り込む: 解答ファイル非破壊・`--refresh` はキャッシュのみ・exit code 規約・標準 `flag` 維持。
- ユーザに委ねるべき分岐だけ `AskUserQuestion` で聞く。慣例的な既定がある選択は決めて理由を 1 行添える。
- 純粋な設計レビュー (ドキュメントを残さず案だけ提示して承認を仰ぐ) で終える場合もあるが、既定の納品は **コミットされた要件定義**。

## 関連

- 実装まで進めるフルワークフロー: `feature` スキル (design の要件をそのまま入力にできる)
- 軽量ワークフロー: `smallwork` スキル
- 振り分け入口: `triage` スキル
- 実装 ⇄ ドキュメントの整合性点検・整頓: `maketidy` スキル
- 要件の雛形: `docs/tools/requirements/003-exercise-abc-contest-meta.md`
- 決定記録: `docs/tools/decisions/` (ADR)
- ルート規約: `CLAUDE.md` (worktree 必須・ディレクトリ規約・解答スタイル)
