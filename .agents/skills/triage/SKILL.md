# triage

この repo の変更ワークフローは 2 本立てになっている:

- **`smallwork`** — typo・微修正・練習解答追加など、軽い変更を「worktree → 編集 → 最小検証 → コミット → ff-merge」で素早く片付ける。
- **`feature`** — `atcoder` CLI に新しい挙動を足す変更を「要件定義 → ロードマップ → 実装 → fixture → 利用ドキュメント → コミット → マージ」のフルセレモニーで進める。

triage はその **入口の交通整理**。着手前に「この変更はどちらの流儀か」を判定し、選んだスキルを起動する。どちらか明白なら triage を経由せず直接そのスキルを呼んでよい — 迷ったとき・中間サイズのとき・依頼者がどちらか指定していないときに使う。

## いつ使うか

- リポジトリへの変更を頼まれたが、`smallwork` と `feature` のどちらで進めるべきか即断できない。
- 変更が「微修正で済む」のか「仕様を伴う新挙動」なのか、規模が読みきれない。
- 依頼にワークフロー指定がなく、まず分類してから着手したい。

## いつ使わないか

- どちらの流儀か最初から明白 → 直接 `smallwork` か `feature` を起動する (triage の一段を挟む必要はない)。
- 読むだけ・調べるだけの作業 (そもそも変更を伴わない)。

## 判定フロー

上から順に評価し、最初に当てはまった行で確定する。

| # | 問い | Yes なら |
|---|---|---|
| 1 | `atcoder` CLI に**新しい挙動**を足すか? (新サブコマンド・新フラグ・新モード・新言語 Runner・新 fetch 経路・新キャッシュ層) | **`feature`** |
| 2 | `abc-todo.md` / `todo.md` のロードマップ項目を 1 つ実装に落とす作業か? | **`feature`** |
| 3 | 設計判断が要る / 影響範囲が複数パッケージにまたがる / 読みきれない | **`feature`** |
| 4 | typo・コメント・文言・docs の軽微な修正か? | **`smallwork`** |
| 5 | 仕様を変えない 1〜数ファイルの局所バグ直し・微調整か? | **`smallwork`** |
| 6 | 練習解答の追加・修正か? (`exercise/`・`abc/`・`arc/`・`awc/`・`adt/`・`dp/`・`tessoku-book/`・`spoj/` 等) | **`smallwork`** |
| 7 | 既存挙動を変えない局所リファクタリングか? | **`smallwork`** |
| 8 | ここまで決まらない (中間サイズ・判断に迷う) | **`feature`** (安全側に倒す) |

**迷ったら `feature`。** 仕様を先に文章化する側に倒すほうが、後で E (本番モード判定) / G (タイマー) が `contest.toml` 等を入力にできるよう前方互換を保てる。「軽いと思ったら重かった」を smallwork で押し切るより、最初から要件を書くほうが安い。

### 境界の目安

- **新フラグを 1 つ足す**: 挙動が増えるので原則 `feature`。ただし既存挙動を変えない出力文言調整レベルなら `smallwork` でもよい — 判断に迷えば `feature`。
- **docs の修正**: リンク・誤字・表現直しは `smallwork`。新フラグ/新サブコマンドに伴う利用ドキュメント整備は `feature` の一部 (単独では発生しない)。
- **ツールに無関係なリポジトリ housekeeping** (skill 整備・設定ファイル・git 周り): 仕様を伴わなければ `smallwork` 流儀でよい (worktree → 編集 → コミット → ff-merge)。

## 判定後にすること

1. 判定結果と理由を 1 行で述べる (例: 「新フラグ追加で仕様を伴うため feature」)。
2. 選んだスキル (`smallwork` または `feature`) を起動し、その手順に従う。
3. 着手後に見立てが外れたら **格上げを優先する**: smallwork で始めて「仕様判断が要る / 大きい」と分かったら、その worktree のまま `feature` の手順に移る。smallwork で押し切らない。逆向き (feature → smallwork) の格下げは基本しない。

## 注意

- triage 自身は変更を加えない。worktree を切るのは振り分け先の `smallwork` / `feature` の手順 0/1 で行う。
- どちらに振っても **worktree は必ず切る** (AGENTS.md 必須)。緩められるのは worktree 以降の重い付帯作業だけ。
- 解答ファイル (ユーザの提出コード) を壊さない、`--refresh` 系はキャッシュのみ対象、exit code 規約 (引数誤り=2 / 実行時失敗=1 / 成功=0) — これらは振り分け先でも共通の不変則。

## 関連

- 軽量ワークフロー: `smallwork` スキル
- フル機能追加ワークフロー: `feature` スキル
- 実装 ⇄ ドキュメントの整合性点検・整頓: `maketidy` スキル
- ツールのスモークテスト: `test-tool` スキル
- ルート規約: `AGENTS.md` (worktree 必須・ディレクトリ規約・解答スタイル)
